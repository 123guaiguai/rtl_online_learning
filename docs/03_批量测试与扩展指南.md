# 批量测试与扩展指南

## 一、批量测试概述

批量测试系统以 `run_batch_experiments.py` 为**唯一入口**，负责对 VerilogEval 数据集中的题目进行流水线式的全自动评测，并统计整体通过率。

### 工作流程

```
run_batch_experiments.py（命令行入口）
        ↓ 读取 VerilogEval/ 目录，扫描所有 Prob*_prompt.txt
        ↓ 按题号分组（easy: 0-49, hard: 50-99）
        ↓ 逐题创建 JSON 配置文件 → configs/config_ProbXXX.json
        ↓ 子进程调用 python generate_verilog.py -c config.json
        ↓ 解析输出目录中的 log.txt，提取 Rank
        ↓ 统计通过率，保存 results_*.json
```

**断点续传**：若某道题的输出目录中已存在含 `Rank of best response` 的 `log.txt`，则自动跳过，无需重新运行。

## 二、API 密钥加载机制

> **批量测试运行时无需任何 API 相关的命令行参数。**

本项目使用 **`api_keys.py` 文件**作为统一的密钥管理机制。`languagemodels.py` 在初始化任何 LLM 客户端时会自动按以下顺序查找并加载该文件：

1. `Experiments/api_keys.py`（项目根目录，**推荐放置位置**）
2. `Experiments/core/api_keys.py`（脚本同级目录，备用）

**配置一次，全局生效**——只要该文件存在且密钥正确，无论是单题运行还是批量测试，均无需再指定任何 API 相关参数。

```python
# Experiments/api_keys.py 文件结构示例
SILICONFLOW_CONFIG = {
    'api_key': 'sk-your-siliconflow-key',
    'base_url': 'https://api.siliconflow.cn/v1'
}

GITHUB_CONFIG = {
    'api_key': 'ghp-your-github-token',
    'base_url': 'https://models.inference.ai.azure.com'
}
```

> 详细配置步骤见 [02_常规使用与部署指南.md](02_常规使用与部署指南.md) 第二节。

## 三、命令行参数说明

```bash
# 基本用法
cd /home/gq/Autochip_workspace/Experiments/core
conda run -n autochip python run_batch_experiments.py [参数]
```

| 参数 | 缩写 | 说明 | 默认值 |
|------|------|------|--------|
| `--group` | `-g` | 测试分组：`easy`（0-49题）/ `hard`（50-99题）/ `all`（全部） | `hard` |
| `--limit` | `-l` | 每组最多测试的题目数（用于快速验证） | 无限制 |
| `--iterations` | `-i` | 每道题的最大迭代次数（覆盖默认配置） | 按组配置 |
| `--candidates` | `-k` | 每次迭代生成的候选代码数量 | `3` |
| `--mixed-models` | `-m` | 强制启用混合模型策略 | 按组配置 |
| `--no-mixed-models` | - | 强制禁用混合模型策略 | - |
| `--gpt-start-iter` | - | 混合模型中第二个模型从第几次迭代介入 | 按组配置 |
| `--experiment-name` | `-n` | 实验名称（用于输出目录命名） | 无 |
| `--output-dir` | `-o` | 批量测试输出根目录 | `outputs/batch_tests` |
| `--overwrite` | - | 覆盖已存在结果（默认为断点续传） | `False` |
| `--api-key` | - | 临时覆盖环境变量，格式：`--api-key KEY VALUE`（不常用） | 无 |
| `--env-file` | `-e` | ~~从 .env 文件加载环境变量~~（已弃用，请使用 `api_keys.py`） | 无 |
| `--dry-run` | - | 预演模式：仅显示配置不实际运行 | `False` |
| `--verbose` | `-v` | 输出详细调试信息 | `False` |

## 四、常用运行示例

### 3.1 快速功能验证（推荐首次使用）

```bash
cd /home/gq/Autochip_workspace/Experiments/core

# 仅测试困难组前 3 题，迭代 1 次，1 个候选（最省时）
conda run -n autochip python run_batch_experiments.py \
    -g hard -l 3 -i 1 -k 1 -n quick_check
```

### 3.2 预演模式（检查配置）

```bash
# --dry-run 不会实际运行，只显示将要执行的配置
conda run -n autochip python run_batch_experiments.py \
    -g hard -l 5 -i 2 -k 2 --dry-run
```

### 3.3 标准批量测试

```bash
# 测试困难组，迭代 2 次，2 个候选，禁用混合模型，实验名 baseline
conda run -n autochip python run_batch_experiments.py \
    -g hard -i 2 -k 2 --no-mixed-models -n baseline
```

### 3.4 混合模型测试

```bash
# 测试困难组，启用混合模型（前 5 次用 Qwen，第 6 次用 gpt-4o-mini）
conda run -n autochip python run_batch_experiments.py \
    -g hard -i 5 -k 3 -m -n mixed_model_exp
```

### 3.5 全组测试

```bash
# 测试全部 100 道题（简单 + 困难），命名为 full_eval
conda run -n autochip python run_batch_experiments.py \
    -g all -i 2 -k 2 -n full_eval
```

### 3.6 后台长时间运行（防止 SSH 断联）

```bash
# 使用 nohup 或 screen 在后台运行
nohup conda run -n autochip python run_batch_experiments.py \
    -g hard -i 5 -k 3 -n long_run \
    > long_run.log 2>&1 &

# 查看运行进度
tail -f long_run.log

# 查看进程是否在运行
ps aux | grep run_batch_experiments
```

## 五、默认分组配置

批量测试系统内置了两个分组的默认实验配置（可通过命令行参数覆盖）：

### 简单组（easy，题号 0-49）

```python
"easy": {
    "range": (0, 50),
    "max_iterations": 2,          # 共尝试 3 次（iter0, iter1, iter2）
    "use_mixed_models": False,
    "model_family": "Siliconflow",
    "model_id": "Qwen/Qwen2.5-Coder-32B-Instruct"
}
```

### 困难组（hard，题号 50-99）

```python
"hard": {
    "range": (50, 100),
    "max_iterations": 5,          # 共尝试 6 次（iter0-iter5）
    "use_mixed_models": True,     # 启用混合模型
    "mixed_models": {
        "model1": {
            "start_iteration": 0,
            "model_family": "Siliconflow",
            "model_id": "Qwen/2.5-Coder-32B-Instruct"
        },
        "model2": {
            "start_iteration": 5,   # 第 6 次迭代切换到 GPT-4o-mini
            "model_family": "ChatGPT",
            "model_id": "gpt-4o-mini",
            "base_url": "https://models.inference.ai.azure.com"
        }
    }
}
```

> **修改默认配置**：直接编辑 `run_batch_experiments.py` 顶部的 `GROUP_CONFIGS` 字典，或通过命令行参数动态覆盖。

## 七、混合模型策略详解

### 6.1 工作原理

混合模型策略允许在不同迭代阶段自动切换不同的 LLM。选择逻辑为：**找到 `start_iteration` 值不超过当前迭代次数的最大模型**。

**示例**（配置了两个模型，切换点为 iter5）：

| 迭代次数 | 使用的模型 |
|---------|-----------|
| 0 ~ 4  | Qwen/Qwen2.5-Coder-32B（Siliconflow） |
| 5+     | gpt-4o-mini（GitHub Models） |

### 6.2 配置要求

使用混合模型时，JSON 配置文件中需满足：
1. `general.mixed-models` 设为 `true`
2. `mixed-models` 节点中必须有一个模型 `start_iteration = 0`
3. 不能有两个模型使用相同的 `start_iteration`
4. 所有 `start_iteration` 必须在 `[0, iterations]` 范围内

### 6.3 成本与效果分析

| 策略 | 优点 | 缺点 |
|------|------|------|
| 单一轻量模型 | 成本低 | 困难题目成功率较低 |
| 单一强模型 | 成功率高 | API 消耗大 |
| 混合策略（推荐） | 成本与效果的最优平衡 | 需要配置两个 API 账户 |

## 十、扩展：添加新的 LLM 支持

如果需要接入新的大语言模型 API，在 `languagemodels.py` 中：

1. 新建一个继承 `AbstractLLM` 的类
2. 实现 `generate(self, conversation, num_candidates)` 方法，返回字符串列表
3. 在 `config_handler.py` 的模型映射表中添加新的 `model_family` 名称

```python
# 示例：添加一个自定义 API
class MyCustomLLM(AbstractLLM):
    def __init__(self, model_id):
        # 初始化客户端
        self.client = ...
    
    def generate(self, conversation, num_candidates=1):
        messages = [{"role": m["role"], "content": m["content"]} 
                    for m in conversation.get_messages()]
        responses = []
        for _ in range(num_candidates):
            resp = self.client.chat(messages)
            responses.append(resp.text)
        return responses
```

# 项目简介与结构说明

## 一、项目背景

本项目基于开源框架 [AutoChip](https://github.com/shailja-thakur/AutoChip) 二次开发而成，核心目标是通过**大语言模型（LLM）自动生成 Verilog 硬件描述代码**，并利用 Icarus Verilog 仿真工具对生成结果进行自动评测，形成"**生成 → 编译 → 仿真 → 反馈 → 再生成**"的闭环迭代流水线。

在原版框架的基础上，本项目进一步添加了：
- 支持断点续传的**参数化批量测试系统**
- 对 Siliconflow、GitHub Models 等多个 API 平台的兼容适配
- 多模型混合迭代策略（不同迭代阶段切换不同模型）
- 对 VerilogEval 测试集的完整流水线评测支持

## 三、核心运行机制

### 3.1 迭代反馈循环

AutoChip 的核心是一个"**生成 → 验证 → 反馈 → 再生成**"的闭环：

```
初始提示词（设计需求）
        ↓
   LLM 生成 Verilog 代码
        ↓
   Icarus Verilog 编译
   ├─ 编译失败 → 将错误日志反馈给 LLM
   └─ 编译成功 ↓
   VVP 仿真测试
   ├─ 仿真不匹配 → 将错误波形信息反馈给 LLM
   └─ 仿真 100% 通过 → 输出结果，Rank = 1.0
        ↓
   未达到最大迭代次数？→ 返回 LLM，进行下一轮迭代
```

每次迭代对应输出目录中的一个 `iter{N}/` 文件夹。`iterations` 参数控制最大迭代轮数，实际执行轮数为 `iterations + 1`（从 iter0 开始计数）。

### 3.2 评分机制（Rank）

每道题的结果用 Rank 值表示：

| Rank 值    | 含义                               |
|------------|------------------------------------|
| **1.0**    | ✅ 完美通过（编译 + 仿真 100% 匹配）|
| 0.0 ~ 0.99 | ⚠️ 部分通过（仿真有部分不匹配）    |
| **-0.5**   | 编译有警告                         |
| **-1**     | ❌ 编译失败（语法错误）             |
| **-2**     | LLM 输出无法提取 Verilog 代码      |

计算公式：`rank = (仿真总样本数 - 不匹配数) / 总样本数`

### 3.3 多候选机制

通过 `num_candidates` 参数，可以让 LLM 在每次迭代中同时生成多个代码版本，每个版本独立编译和仿真，系统自动选择 Rank 最高（同 Rank 时选最短）的代码进入下一轮迭代。

```
iter0/
├── response0/   ← 候选 0（rank=-1，编译失败）
├── response1/   ← 候选 1（rank=0.95，部分通过）
└── response2/   ← 候选 2（rank=1.0，完美）✅ 被选中
```

### 3.4 混合模型策略

支持在不同迭代阶段切换不同的 LLM 模型，以平衡成本与效果：

- **前期迭代**（iter0-4）：使用轻量级模型（如 Qwen/Qwen2.5-Coder-32B）
- **后期迭代**（iter5+）：切换到更强的模型（如 gpt-4o-mini）

## 四、关键文件说明

### 核心执行文件

| 文件 | 说明 |
|------|------|
| `core/generate_verilog.py` | **单题入口**，接收 JSON 配置，运行一道题的完整迭代流程 |
| `core/run_batch_experiments.py` | **批量入口**，内置全部批量测试逻辑，唯一批量测试入口 |
| `core/analyze_batch_results.py` | 批量测试结束后，汇总 log.txt 生成通过率报告 |
| `core/verilog_handling.py` | 主迭代循环、代码提取、iverilog 编译与仿真调用 |
| `core/languagemodels.py` | LLM 接口实现（支持 OpenAI / Claude / Gemini 等）及 Rank 计算 |
| `core/config_handler.py` | 解析 JSON 配置文件，验证混合模型配置合法性 |
| `core/conversation.py` | 维护多轮对话历史，管理 system / user / assistant 消息 |

### 数据集目录（`VerilogEval/`）

批量测试的题目来源，文件命名规则：
- `Prob001_zero_prompt.txt`：题目描述（作为 LLM 的提示词输入）
- `Prob001_zero_test.sv`：仿真测试台（用于编译和仿真验证）
- `Prob001_zero_ref.sv`：参考模型（黄金答案，编译时一并加入）

### 输出目录（`core/outputs/`）

每次批量测试生成一个以时间戳命名的子目录：

```
outputs/batch_tests/
└── my_exp_hard_20260506_120000/    ← 实验目录
    ├── results_hard.json            ← 本次实验汇总（JSON）
    └── Prob050_xxx/                 ← 每道题的输出
        ├── log.txt                  ← 主日志（含最终 Rank）
        └── iter0/
            └── response0/
                ├── TopModule.sv     ← 生成的代码
                └── TopModule.vvp   ← 编译产物
```

```

        ↓ (读取提示词)

        ↓ (创建配置文件)

        ↓ (启动子进程)
generate_verilog.py
        ↓ → config_handler.py（解析配置）
        ↓ → conversation.py（初始化对话）
        ↓ → languagemodels.py（调用 LLM API）
        ↓ → verilog_handling.py（迭代循环、编译、仿真）
        ↓
outputs/batch_tests/exp_xxx/Prob001_zero/log.txt（结果写入）
        ↓
analyze_batch_results.py（汇总统计）
```
